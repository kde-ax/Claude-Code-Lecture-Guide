## 한 줄 요약

API의 정의(두 소프트웨어 간 통신 계약), 작동 방식(클라이언트-서버 + 요청/응답), 4종 아키텍처(SOAP/RPC/WebSocket/REST), 사용 범위(프라이빗/퍼블릭/파트너/복합), 보안(인증 토큰/API 키), 운영 도구(API 게이트웨이)를 입문자 눈높이로 정리한 AWS 공식 문서.

## 핵심 takeaway

- **API = 두 소프트웨어 구성 요소가 약속된 형식으로 통신하게 해주는 메커니즘.** 인터페이스는 서비스 계약, API 문서는 그 계약서다.
- **작동 모델**: 요청을 보내는 쪽이 클라이언트, 응답을 보내는 쪽이 서버. 날씨 앱이 기상청 API를 호출하는 게 전형적인 예시.
- **4종 아키텍처**: SOAP(XML 기반, 구식·비유연) / RPC(원격 함수 호출) / WebSocket(JSON 기반 양방향) / **REST**(현대 표준, HTTP+무상태).
- **REST 핵심**: GET/PUT/DELETE 같은 함수 집합 + HTTP + **무상태**(서버가 클라이언트 상태 저장 안 함).
- **사용 범위 4종**: 프라이빗(사내) / 퍼블릭(누구나) / 파트너(B2B 권한) / 복합(여러 API 결합).
- **보안 2축**: 인증 토큰(사용자 신원 확인, 더 안전) vs API 키(앱 식별 + 사용량 추적용).
- **API 게이트웨이** = 여러 API 앞단에서 인증·트래픽 관리·통계 일괄 처리. AWS의 Amazon API Gateway가 대표 사례.
- **GraphQL** = REST의 대안. 단일 엔드포인트로 클라이언트가 "필요한 데이터만" 골라 요청.
- **비즈니스 가치 4가지**: 통합(기존 코드 재사용) / 혁신(부분 변경만으로 빠른 배포) / 확장(다양한 플랫폼 지원) / 유지관리 용이(시스템 간 격리).

## 주요 인용

> "API는 정의 및 프로토콜 집합을 사용하여 두 소프트웨어 구성 요소가 서로 통신할 수 있게 하는 메커니즘입니다."

> "REST API의 주된 특성은 무상태입니다. 무상태는 서버가 요청 간에 클라이언트 데이터를 저장하지 않음을 의미합니다."

> "인증 토큰은 사용자에게 API 호출을 수행할 수 있는 권한을 부여하는 데 사용됩니다. … API 키는 API를 호출하는 프로그램 또는 애플리케이션을 확인합니다. … API 키는 토큰만큼 안전하지 않지만 사용량에 대한 데이터를 수집하기 위해 API 모니터링을 허용합니다."

## 한계 / 비판 지점

- 후반부가 Amazon API Gateway, AWS AppSync 영업으로 넘어간다 — 일반론과 분리해서 읽기.
- "모든 웹 서비스는 API이지만 모든 API가 웹 서비스는 아니다" 같은 문장은 입문자에게 더 헷갈릴 수 있어 위키 본문에서는 더 친절하게 풀어 쓸 필요.
- API 키 vs 인증 토큰의 차이가 보안 관점에서 너무 압축되어 있어 실제 운영(OAuth 2.0, JWT 등)에 닿지 않는다.
