## 정의

API(Application Programming Interface)는 **두 소프트웨어가 약속된 형식으로 대화하게 해주는 통신 메커니즘**이다 [[20260504_AWS-API란]]. 인터페이스는 **서비스 계약**이고, API 문서는 그 계약서다.

핵심 단어 풀이:
- **Application** = 어떤 기능을 가진 소프트웨어
- **Programming Interface** = 프로그램끼리 정보를 주고받는 약속된 통로

## 핵심 메커니즘

API의 작동은 [[클라이언트-서버]] 모델로 설명된다.

```
[클라이언트] → 요청 (request) → [서버]
[클라이언트] ← 응답 (response) ← [서버]
```

- **요청을 보내는 쪽 = 클라이언트** (예: 휴대폰의 날씨 앱)
- **응답을 보내는 쪽 = 서버** (예: 기상청의 날씨 데이터베이스)

날씨 앱이 화면을 그릴 때, 앱은 직접 기상 위성을 보지 않는다. 기상청의 API에 "오늘 서울 날씨 알려줘"라고 요청하고, 받은 응답을 화면에 표시한다.

## 일상 비유

식당에서 손님이 주방에 직접 못 들어가지만 **웨이터를 통해 주문**할 수 있는 것과 같다.
- 손님 = 클라이언트 (앱)
- 웨이터 = API (정해진 메뉴판 = 요청 형식)
- 주방 = 서버 (실제 요리하는 곳)

손님은 주방이 어떻게 돌아가는지 몰라도 된다. 메뉴판(API 문서)만 보면 "스파게티 곱빼기" 주문할 수 있고, 받을 수 있다.

## API 종류

### 아키텍처별 4종
- [[REST-API]] — 현대 표준, HTTP 기반, 무상태
- [[SOAP]] — XML 기반, 구식·견고
- [[RPC]] — 원격 함수 호출
- [[WebSocket]] — 양방향 실시간 통신

### 사용 범위별 4종
- **프라이빗 API**: 회사 내부에서만 사용
- **퍼블릭 API**: 누구나 사용 가능 (예: Twitter API, OpenAI API)
- **파트너 API**: B2B 권한 부여된 외부 개발자만
- **복합 API**: 여러 API를 조합해 한 번에 호출

## 비즈니스 가치

[[20260504_AWS-API란]]에서 정리하는 4가지:
1. **통합** — 새 앱을 기존 시스템에 붙일 수 있음 (코드 재작성 X)
2. **혁신** — API 수준에서만 변경하면 빠른 배포 가능
3. **확장** — 같은 API를 웹·iOS·Android 등 여러 플랫폼에서 호출
4. **유지관리 용이** — 시스템 간 격리. 한쪽이 바뀌어도 다른 쪽 영향 최소
