# LLM-Wiki 작업 지침 (CCFM AX팀 / 강동이)

> 이 파일은 카파시(Andrej Karpathy)의 **LLM-Wiki 패턴**을 우리 팀 작업 환경에 맞춰 적용한 스키마다.
> 사용자(강동이)는 출처를 큐레이션하고 질문을 던지며, **위키의 실제 작성·유지보수는 Claude(LLM)가 전담**한다.

---

## 0. 핵심 원칙 (Why)

전통적인 RAG는 매 질문마다 원본 문서에서 청크를 검색해 답을 생성한다. 지식이 누적되지 않는다.
이 위키는 다르다. **새 소스가 들어오면 LLM이 즉시 통합 — 엔티티 페이지 갱신, 개념 정리 보강, 모순 표시, 인덱스/로그 업데이트** — 까지 한 번에 끝낸다.
질의 시점이 아니라 **수집 시점에 한 번 컴파일**해서, 위키 자체가 시간이 갈수록 풍부해지는 영속 자산이 되게 한다.

> 사용자의 일: 소스 큐레이션 + 좋은 질문
> Claude의 일: 요약 / 교차참조 / 정리 / 모순 추적 / 색인 관리 — **귀찮은 잡일 전부**

---

## 1. 3계층 아키텍처

```
llm-wiki/
├── raw/        ← 불변 원본 (LLM은 읽기만, 수정 금지)
├── wiki/       ← LLM이 생성·유지하는 마크다운 지식베이스
├── Clippings/  ← Obsidian Web Clipper 자동 저장 폴더 (raw 와 동급으로 취급)
└── CLAUDE.md   ← 이 파일 (스키마)
```

각 폴더에는 별도의 `CLAUDE.md`가 있어 폴더별 세부 규칙을 정의한다. 전체 규칙이 충돌하면 **하위 폴더 규칙이 우선**한다.

---

## 2. 디렉터리별 역할

| 폴더 | 누가 쓰는가 | 무엇을 담는가 |
|------|------------|---------------|
| `raw/` | 사용자 (드롭) | 불변 원본 — 기사, PDF, 회의록, 트랜스크립트, 이미지 |
| `Clippings/` | Obsidian Web Clipper | 웹에서 클리핑한 마크다운 (raw 의 일부로 간주) |
| `wiki/` | **Claude 전담** | 엔티티·개념·소스 요약·종합 분석, 인덱스, 로그 |

## 3. 표준 워크플로우

### 3.1 Ingest (소스 수집·통합)

사용자가 `raw/` 또는 `Clippings/`에 새 파일을 떨어뜨리고 "넣어줘"라고 말하면:

1. **읽기**: 원본을 읽고 핵심 takeaway 3~5개를 사용자에게 먼저 보고
2. **소스 페이지 작성**: `wiki/sources/[제목]_YYYYMMDD.md` 생성 (요약 + 핵심 인용 + 출처 메타)
3. **엔티티/개념 갱신**: 등장한 인물·조직·제품·개념별로 `wiki/entities/`, `wiki/concepts/` 페이지를 생성하거나 갱신 — 한 소스가 보통 10~15개 페이지를 건드린다
4. **모순 플래그**: 기존 페이지의 주장과 어긋나면 `> ⚠️ 모순:` 블록으로 명시 (덮어쓰지 말 것)
5. **인덱스 갱신**: `wiki/index.md`에 새/변경 페이지 한 줄씩 추가
6. **로그 추가**: `wiki/log.md` 맨 아래에 한 줄 (포맷은 §5 참고)

> 한 번에 한 소스씩 처리하는 걸 기본으로 한다. 사용자가 "여러 개 한꺼번에"라고 명시하면 배치 처리.

### 3.2 Query (질의)

사용자가 위키에 질문하면:

1. **인덱스 우선**: `wiki/index.md`를 먼저 읽고 관련 페이지 후보를 추린다
2. **드릴다운**: 관련 페이지를 읽고 합성한 답변을 작성 — 모든 주장에 `[[페이지명]]` 위키링크로 출처 표시
3. **재투입 제안**: 답변이 일회성으로 휘발될 가치 이상이면 "이거 `wiki/synthesis/`에 페이지로 저장할까?"라고 먼저 묻고 OK 받으면 저장
4. 답변 형식은 질문에 맞게: 마크다운 / 비교 표 / Marp 슬라이드 / 차트 등

### 3.3 Lint (위키 점검)

사용자가 "린트", "위키 점검", "건강 체크"라고 하면:

- 모순되는 주장 (서로 다른 페이지 간)
- 더 새로운 소스로 대체된 stale claim
- inbound 링크 0개인 고아 페이지
- 본문에서 자주 언급되지만 자체 페이지가 없는 개념
- 누락된 교차 참조
- 웹서치로 메울 만한 데이터 갭

순서대로 보고하고, 사용자가 승인한 항목만 실제 수정한다.

## 4. 페이지 작성 규칙

### 4.1 Frontmatter (모든 wiki 페이지 필수)

```yaml
---
title: 페이지 제목
type: entity | concept | source | synthesis
tags: [태그1, 태그2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [[source-page-1]], [[source-page-2]]
---
```

- `type`은 **entity / concept / source / synthesis** 4종 중 하나
- `updated`는 페이지를 수정할 때마다 갱신
- `sources`는 이 페이지의 주장을 뒷받침하는 raw 소스의 위키링크 목록 (Provenance)

### 4.2 본문 작성 원칙

- **한국어로 작성** (CCFM 팀 공통 원칙)
- 주장 옆에는 출처 링크 — 예: `n8n은 셀프호스트 가능한 워크플로우 엔진이다 [[2026-04-12_n8n공식문서]]`
- 다른 페이지로 교차 참조는 항상 위키링크: `[[페이지명]]`
- 모순/미확인은 명시 — `> ⚠️ 모순: [[페이지A]]는 X라 하지만 [[페이지B]]는 Y라고 함`
- 추측/미확인 정보는 `> ❓ TBD:` 블록
- 코드/명령어는 ``` 펜스, 짧은 명령어는 인라인 백틱
- 표는 | 마크다운 표

### 4.3 파일명 규칙

| 폴더 | 파일명 패턴 | 예시 |
|------|-------------|------|
| `wiki/entities/` | `[엔티티명].md` | `n8n.md`, `강동이.md` |
| `wiki/concepts/` | `[개념명].md` | `LLM-Wiki패턴.md` |
| `wiki/sources/` | `YYYYMMDD_[제목].md` | `20260504_카파시-LLM-Wiki.md` |
| `wiki/synthesis/` | `[주제].md` | `RAG-vs-위키패턴.md` |

공백은 `-` 또는 `_`로. 한국어 그대로 OK (Windows/Obsidian 호환).

## 5. 인덱스와 로그

### 5.1 `wiki/index.md` (콘텐츠 카탈로그)

**모든 페이지 한 줄씩**, 카테고리별로 묶어 관리. 매 ingest마다 갱신.

```markdown
## 엔티티
- [[n8n]] — 셀프호스트 워크플로우 자동화 엔진 (소스 3개)
- [[강동이]] — CCFM AX팀 AI 기획자

## 개념
- [[LLM-Wiki패턴]] — 카파시가 제안한 영속 위키 + LLM 유지보수 패턴

## 소스
- [[20260504_카파시-LLM-Wiki]] (2026-05-04)

## Synthesis
(없음)
```

### 5.2 `wiki/log.md` (시간순 활동 기록)

**append-only**. 형식은 grep 가능하게 고정:

```markdown
## [2026-05-04] ingest | 카파시 LLM-Wiki gist
- 새 소스 페이지: [[20260504_카파시-LLM-Wiki]]
- 신규 개념: [[LLM-Wiki패턴]], [[Memex]]
- 갱신 엔티티: [[Andrej-Karpathy]]

## [2026-05-04] lint
- 고아 페이지 0건, 모순 0건
```

prefix는 항상 `## [YYYY-MM-DD] {ingest|query|lint|setup} | 짧은제목` 패턴. 이렇게 하면 `grep "^## \[" log.md | tail -10` 로 최근 10건 바로 볼 수 있다.

## 7. 사용자 협업 규칙

- 사용자가 "넣어줘" / "ingest" → §3.1
- 사용자가 "물어볼게" / 일반 질문 → §3.2
- 사용자가 "린트" / "점검" → §3.3
- 사용자가 "이거 페이지로 만들어" → 즉시 wiki에 페이지 생성 후 인덱스/로그 갱신
- 사용자가 위키 페이지를 직접 수정한 흔적이 있으면 (frontmatter `updated` 미스매치 등) **덮어쓰지 말고 먼저 알리기**

## 8. 깃 / Obsidian 통합

- 이 폴더는 git 저장소이자 Obsidian vault다 — 모든 변경은 평문 마크다운이라 자동 버전관리됨
- Obsidian 그래프 뷰에서 [[위키링크]]가 그래프 엣지가 됨 — 가능한 한 풍부하게 링크
- 이미지를 다운로드해 두면 LLM이 직접 볼 수 있다 (`raw/assets/`)
- Obsidian Web Clipper의 기본 저장 위치인 `Clippings/`는 그대로 두고 raw 동급으로 취급. 정리가 필요하면 `raw/articles/`로 이동 가능
